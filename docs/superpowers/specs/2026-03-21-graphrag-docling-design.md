# 회계기준서 GraphRAG 아키텍처 설계서

> Docling + EdgeQuake + LangGraph 기반 회계기준서 RAG 시스템
>
> 작성일: 2026-03-21
> 상태: 초기 설계 (반복 업데이트 예정)
>
> **이 문서는 기존 `회계기준서_RAG_Architecture.md`를 대체합니다.**
> 변경 사유: LlamaIndex → Docling 전환, Milvus → EdgeQuake(PostgreSQL 통합) 전환, GraphRAG 도입

---

## 1. 프로젝트 개요

### 목적
K-IFRS 회계기준서(70개, 평균 250페이지)를 대상으로 한 GraphRAG 시스템 구축.
회계사/감사인이 실무에서 정확한 기준서 조항과 상호참조를 검색할 수 있도록 한다.

### 대상 사용자
- **1차**: 회계사/감사인 — "이 거래에 적용할 기준서와 관련 예외 조항을 찾아줘"
- **2차**: 개발자 포트폴리오 — 기술 깊이 시연

### 대상 문서
- 형식: PDF, HTML, Word, HWP (혼합)
- 규모: 70개 문서, 평균 250페이지 (총 ~17,500페이지)
- 특성: 계층 구조(장/절/문단), 상호참조, 테이블, 수식, 조건부 예외 조항

## 2. 전체 아키텍처

```
[회계기준서 문서들 (PDF/HTML/Word/HWP)]
        │
        ▼
┌───────────────────┐
│     Docling        │  정밀 파싱 (테이블/수식/계층구조)
│  (HWP → PDF 변환)  │  HWP는 pyhwp/LibreOffice로 사전 변환
└───────┬───────────┘
        │ 구조화된 텍스트 + 메타데이터
        ▼
┌───────────────────────────────┐
│  Document Preprocessor         │
│  ├─ 계층 보존 청킹            │  문단/절 경계 기반 청킹
│  ├─ 회계 도메인 엔티티 매핑    │  EdgeQuake 엔티티 타입 커스터마이징
│  ├─ 상호참조 어노테이션        │  "제1028호 문단15" → 관계 힌트
│  └─ 메타데이터 추출            │  기준서번호, 시행일, 적용범위
└───────┬───────────────────────┘
        │
        ▼
┌───────────────────────────────┐
│       EdgeQuake                │
│  ┌─ Apache AGE (그래프DB)     │  엔티티/관계 자동 추출 + 저장
│  ├─ pgvector (벡터DB)         │  청크 임베딩 + 유사도 검색
│  ├─ 6가지 쿼리 모드           │  아래 상세 참조
│  └─ 커뮤니티 감지             │  주제별 기준서 클러스터링
└───────┬───────────────────────┘
        │ 검색 결과 (그래프 + 벡터)
        ▼
┌───────────────────────────────┐
│       LangGraph                │  Agentic 오케스트레이션
│  ├─ Query Rewriter            │  자연어 → 회계 용어 변환
│  ├─ Query Router              │  쿼리 유형별 모드 선택 (별도 단계)
│  ├─ CRAG Gate                 │  품질 게이트 (검색 충분한가?)
│  └─ Answer Generator          │  구조화 응답 + 인용
└───────┬───────────────────────┘
        │
        ▼
┌───────────────────┐
│  PydanticAI        │  타입 안전 구조화 출력 (인용, 페이지번호)
└───────────────────┘
```

### 핵심 설계 결정

- **Docling** — 회계 문서의 테이블/수식을 정밀 파싱 (EdgeQuake 자체 pdfium보다 우수)
- **EdgeQuake** — 그래프 구축 + 벡터 저장 + 하이브리드 검색을 하나의 엔진으로 통합
- **PostgreSQL 통합** — Apache AGE(그래프) + pgvector(벡터)를 단일 DB로 운영
- **LangGraph** — EdgeQuake 위에서 CRAG 품질 게이트와 멀티스텝 에이전트 제어
- **PydanticAI** — 최종 응답의 구조화 출력

### 그래프 DB 관련 참고
- Apache AGE는 Neo4j 대비 그래프 전용 인덱싱, 탐색 최적화가 부족
- 현재 규모(70개 문서, 1-3홉 탐색)에서는 충분
- 향후 그래프 검색 성능이 부족하면 Neo4j로 교체 검토 가능 (부분적 Cypher 호환, 쿼리 감사 및 적응 필요)

## 3. Docling 파싱 파이프라인

### 왜 Docling인가
- EdgeQuake의 pdfium은 단순 텍스트 추출 → 회계 문서의 테이블, 수식, 계층 구조를 잃음
- Docling은 문서의 레이아웃을 이해하고 테이블을 구조화된 데이터로 변환

### HWP 처리 전략
Docling은 HWP를 네이티브 지원하지 않으므로 사전 변환 필요:
- **1순위**: `pyhwp` 라이브러리로 HWP → PDF 변환 후 Docling 처리
- **2순위**: LibreOffice CLI (`libreoffice --convert-to pdf`)로 변환
- **폴백**: 수동 PDF 변환 (소수의 변환 실패 케이스)

### 청킹 전략

회계기준서의 계층 구조를 보존하는 구조 기반 청킹:

```
기준서 1116 (루트)
├─ 제1장 목적 (청크 단위: 장)
├─ 제5장 사용권자산
│  ├─ 문단 31 (청크 단위: 문단)
│  ├─ 문단 32
│  └─ 문단 33
└─ 부록 적용사례
```

- **기본 청크 단위**: 문단 (paragraph) — Docling의 구조 파싱 결과 기반
- **청크 크기**: 목표 500-1000 토큰, 문단이 너무 길면 문장 경계에서 분할
- **컨텍스트 보존**: 각 청크에 상위 계층 경로를 메타데이터로 포함
  - 예: `{기준서: "1116", 장: "제5장 사용권자산", 문단: "31"}`
- **오버랩**: 인접 문단 간 1문장 오버랩으로 문맥 연결 유지

### 파이프라인 흐름

```
[원본 문서 (PDF/HTML/Word/HWP)]
        │
        ├─ HWP → pyhwp/LibreOffice → PDF 변환
        │
        ▼
┌───────────────────┐
│     Docling        │
│  ├─ PDF 파싱       │  레이아웃 분석 + OCR
│  ├─ 테이블 추출    │  → 구조화된 테이블 (Markdown/JSON)
│  ├─ 수식 인식      │  → LaTeX 표현
│  └─ 계층 구조 보존  │  장 > 절 > 문단 트리
└───────┬───────────┘
        │ DoclingDocument 객체
        ▼
┌───────────────────┐
│  전처리기           │
│  ├─ 구조 기반 청킹  │  문단 경계 기반, 계층 메타데이터 부착
│  ├─ 상호참조 추출   │  정규식: "제\d{4}호", "문단\s*\d+"
│  ├─ 메타데이터 추출  │  기준서번호, 시행일, 적용범위
│  └─ 텍스트 변환     │  EdgeQuake REST API 페이로드(텍스트+메타데이터)로 변환
└───────┬───────────┘
        │
        ▼
┌───────────────────┐
│  EdgeQuake 인제스트  │  REST API를 통해 문서 투입
│  ├─ 엔티티 추출     │  LLM 기반 (회계 도메인 프롬프트)
│  ├─ 관계 추출       │  + 전처리기의 상호참조 힌트 반영
│  └─ 그래프+벡터 저장 │  AGE + pgvector
└───────────────────┘
```

### 실패 처리
- Docling 파싱 실패 → 로그 기록 + 해당 문서 스킵, 나머지 계속 처리
- 부분 인제스트 실패 → 실패 문서 목록 저장, 재시도 가능하도록 멱등성 보장
- LLM API 타임아웃 → 지수 백오프 재시도 (최대 3회)

## 4. 검색 및 응답 파이프라인

### EdgeQuake 쿼리 모드 활용

| 모드 | 설명 | 회계 도메인 용도 |
|------|------|-----------------|
| **local** | 엔티티 중심 + 이웃 탐색 | 특정 기준서/조항 검색 |
| **global** | 커뮤니티 기반 의미 검색 | 주제 전반 탐색 ("리스 관련 기준서 전부") |
| **hybrid** | local + global 결합 | 복합 질의 (기본값) |
| **naive** | 벡터 유사도만 사용 | 단순 키워드 검색, 폴백용 |
| **mix** | 벡터/그래프 가중 블렌딩 | 향후 튜닝용으로 예비 |
| **bypass** | 검색 없이 LLM 직접 | 사용 안 함 (인용 필수이므로) |

### 파이프라인 흐름

```
[사용자 질의]
        │
        ▼
┌─────────────────────────────────┐
│  LangGraph 오케스트레이션        │
│                                  │
│  ① Query Rewriter               │
│     자연어 → 회계 용어 변환       │
│     출력: {rewritten_query, ...} │
│        │                         │
│        ▼                         │
│  ② Query Router (별도 단계)      │
│     쿼리 유형 분석 → 모드 선택    │
│     출력: {mode: "local"|        │
│            "global"|"hybrid"}    │
│        │                         │
│        ▼                         │
│  ③ EdgeQuake 검색                │
│     선택된 모드로 검색 실행       │
│        │                         │
│        ▼                         │
│  ④ CRAG 품질 게이트              │
│     검색 결과 평가:               │
│     ├─ CORRECT → ⑥으로           │
│     ├─ AMBIGUOUS → ⑤ 보충 검색   │
│     └─ WRONG → ⑤ 재검색          │
│        │                         │
│        ▼                         │
│  ⑤ 보충 검색 (필요 시, 최대 2회) │
│     ├─ 쿼리 리라이팅 후 재검색    │
│     ├─ 다른 모드로 재시도         │
│     └─ 그래프 탐색 확장 (홉 증가) │
│        │                         │
│        ▼                         │
│  ⑥ Answer Generator             │
│     검색된 컨텍스트 → LLM 응답    │
└────────┬────────────────────────┘
         │
         ▼
┌─────────────────────────────────┐
│  PydanticAI 구조화 출력          │
│  {                               │
│    "answer": "...",              │
│    "citations": [                │
│      {"기준서": "1116",          │
│       "문단": "31-33",           │
│       "내용": "..."}             │
│    ],                            │
│    "related_standards": [...],   │
│    "confidence": 0.92            │
│  }                               │
└─────────────────────────────────┘
```

### 핵심 설계 결정
- **Query Rewriter와 Query Router를 분리** — 리라이팅은 LLM 호출, 라우팅은 규칙 기반 또는 경량 분류
- **CRAG 루프는 최대 2회 재시도** — 무한 루프 방지, 2회 실패 시 부분 결과 + 안내 메시지 반환
- **bypass 모드 사용 안 함** — 인용 없는 답변을 허용하지 않으므로
- **PydanticAI로 응답 스키마 강제** — 인용 없는 답변이 나올 수 없도록 타입 레벨에서 보장

## 5. 회계 도메인 커스터마이징

> 초기 설계이며 반복적으로 업데이트 예정

### 엔티티 타입

| EdgeQuake 기본 | 회계 도메인 매핑 | 예시 | 구분 방법 |
|---------------|-----------------|------|----------|
| concept | **회계기준** (standard) | K-IFRS 1116, K-IFRS 1028 | 프로퍼티: `subtype: "standard"` |
| concept | **회계개념** (accounting_concept) | 사용권자산, 감가상각, 공정가치 | 프로퍼티: `subtype: "concept"` |
| event | **회계처리** | 최초인식, 후속측정, 제거 | - |
| organization | **규제기관** | 금융감독원, IASB | - |
| product | **재무제표항목** | 재무상태표, 손익계산서 | - |
| technology | 사용 안 함 | - | - |
| location | 사용 안 함 | - | - |

> `concept` 타입의 이중 매핑은 `subtype` 프로퍼티로 구분하여 그래프 쿼리 시 필터링 가능

### 관계(엣지) 타입

```
[기준서 1116] ──참조──→ [기준서 1028]
[사용권자산] ──적용──→ [감가상각]
[리스부채]  ──예외──→ [단기리스]
[최초인식]  ──후속──→ [후속측정]
[문단 31]   ──소속──→ [제5장 사용권자산]
```

주요 관계 유형:
- **참조** — 기준서 간 상호참조
- **소속** — 계층 구조 (장 > 절 > 문단)
- **적용** — 개념 → 회계처리 연결
- **예외** — 조건부 예외 조항
- **후속** — 처리 순서/단계

### EdgeQuake 프롬프트 커스터마이징
- 엔티티 타입을 위 매핑으로 제한
- "제XXXX호", "문단 XX" 패턴을 우선 엔티티로 추출하도록 지시
- 예외 조항("다만", "제외하고", "적용하지 아니한다")을 관계로 명시 추출

## 6. 기술 스택 및 모델 선택

### 인프라

| 레이어 | 기술 | 역할 |
|--------|------|------|
| 문서 파싱 | **Docling** | PDF/HTML/Word → 구조화된 텍스트 |
| HWP 변환 | **pyhwp / LibreOffice** | HWP → PDF 사전 변환 |
| Graph-RAG 엔진 | **EdgeQuake** (Rust) | 그래프 구축 + 벡터 저장 + 하이브리드 검색 |
| 그래프 DB | **Apache AGE** (PostgreSQL 확장) | 엔티티/관계 저장, Cypher 쿼리 |
| 벡터 DB | **pgvector** (PostgreSQL 확장) | 임베딩 저장, 유사도 검색 |
| 오케스트레이션 | **LangGraph** (Python) | Agentic 워크플로우, CRAG 루프 |
| 구조화 출력 | **PydanticAI** (Python) | 타입 안전 응답 생성 |
| 패키지 관리 | **uv** | Python 의존성 |
| 컨테이너 | **Docker Compose** | EdgeQuake + PostgreSQL 통합 배포 |

### LLM 모델 선택

| 태스크 | 모델 (1순위) | 모델 (대안) | 비고 |
|--------|-------------|------------|------|
| 엔티티 추출 (배치) | GPT-4o-mini | Ollama (로컬) | 비용 효율 우선, ~17,500페이지 처리 |
| 쿼리 리라이팅 | GPT-4o-mini | - | 실시간, 저비용 |
| CRAG 평가 | GPT-4o-mini | - | 간단한 분류 태스크 |
| 응답 생성 | GPT-4o | Claude 3.5 Sonnet | 정확도 우선 |

### 임베딩 모델

한국어 회계 텍스트에 적합한 모델 선택 필요:
- **1순위**: `multilingual-e5-large` — 한국어 성능 우수, 오픈소스
- **대안**: OpenAI `text-embedding-3-small` — API 기반, 간편
- 선택 기준: 한국어 회계 용어 유사도 벤치마크 후 결정 (구현 단계에서 비교 테스트)

### 비용 추정 (엔티티 추출 배치)
- ~17,500페이지 × ~500 토큰/페이지 = ~8.75M 입력 토큰
- GPT-4o-mini 기준: ~$1.3 (입력) + 출력 비용
- Gleaning(2차 추출) 포함 시: ~$3-5 총 비용

## 7. 개발 순서

1. Docker로 EdgeQuake + PostgreSQL 환경 구축
2. Docling 파싱 파이프라인 구현 (HWP 변환 포함)
3. EdgeQuake 도메인 커스터마이징 (엔티티/관계 타입, 프롬프트)
4. LangGraph 오케스트레이션 구현 (Query Rewriter → Router → EdgeQuake → CRAG → Answer)
5. PydanticAI 응답 스키마 구현
6. 테스트 및 평가

---

## 8. 평가 전략

> 구현 단계에서 상세화 예정

- **테스트 데이터셋**: 회계사 실무 질의 40-50개 수작업 작성
- **Ground Truth**: 각 질의에 대해 정답 기준서/문단 번호를 수동 태깅
- **평가 지표**:
  - Retrieval Precision/Recall — 올바른 기준서/문단을 검색했는가
  - Citation Accuracy — 인용이 실제 원문과 일치하는가
  - Faithfulness — 응답이 검색된 컨텍스트에 근거하는가
  - 상호참조 추적 정확도 — 관련 예외 조항을 빠뜨리지 않았는가
